# Epheon 开发流程

这份文档只定义一件事：当代码、设计和任务发生变化时，应该先改哪一层文档。

## 一、流程总览

Epheon 当前使用下面这条主链路：

```txt
RFC
  ↓
plans
  ↓
tasks
  ↓
code / tests
  ↓
reviews
```

## 二、各层职责

### `docs/rfcs/`

负责：

- 架构原则
- 包边界
- 公共 API 语义
- 数据模型与长期设计决策

什么时候先改它：

- 变更公共 API 承诺
- 新增或删除 package 边界
- 修改时间模型、参考系、星历协议、历法规则等核心语义

### `docs/plans/`

负责：

- 阶段目标
- 阶段边界
- 进入条件
- 完成判定

什么时候改它：

- 某个阶段的目标范围发生变化
- 某个阶段的“暂缓 / 主线”判断发生变化
- 需要重排阶段顺序或阶段职责

### `docs/tasks/`

负责：

- 具体任务拆分
- 任务状态
- 依赖关系
- 关联代码与阶段执行入口

什么时候改它：

- 开始一个任务
- 完成一个任务
- 拆分或合并一个任务
- 调整执行顺序

规则：

- 任务状态只在 `docs/tasks/` 维护。
- `docs/plans/` 不重复维护任务级状态。

### `code / tests`

负责：

- 实际实现
- 测试验证
- fixture 与数据快照

什么时候改它：

- 任务已经在 `docs/tasks/` 中明确
- 所需边界已经在 RFC 或 plans 中稳定

### `docs/reviews/`

负责：

- 阶段性审查
- RFC 与代码的一致性记录
- 收口结论与残余问题

什么时候改它：

- 一轮阶段收口完成后
- 做专项审查或一致性审计时

## 三、最小工作规则

1. 改边界或公共 API：先改 RFC。
2. 改阶段范围：改 plans。
3. 开始或完成具体任务：改 tasks。
4. 落实现与测试：改 code / tests。
5. 做阶段收口：补 reviews。

1. `docs/plans/0007-post-minimum-roadmap.md`
2. `docs/tasks/0007-post-minimum-stabilization-tasks.md`
