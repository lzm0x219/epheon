# Epheon 第五阶段计划：验证体系

本文对应 `docs/rfcs/0001-architecture.md` 的“第五阶段：验证体系”。本文只维护阶段目标、边界和完成判定；任务状态、当前代码产出与执行顺序统一维护在 `docs/tasks/0005-stage-5-verification-tasks.md`。

```txt
证明计算结果可信
```

## 二、执行入口

本阶段的任务状态、关联代码与验证闭环见：

```txt
docs/tasks/0005-stage-5-verification-tasks.md
```

## 三、当前阶段的执行顺序

第五阶段不需要铺很大，先补最小验证闭环：

1. 先补中国历法相关 fixture。
2. 再让 `conformance/` 至少跑通一条跨实现核对链路。
3. 最后给 `benchmarks/` 补主路径基准。

## 四、当前应实现的最小切片

优先实现：

1. 新增中国历法 fixture，例如农历月序、闰月年与已知历日样例。
2. 在 `conformance/` 中加入至少一个可以独立运行的核对入口。
3. 在 `benchmarks/` 中加入年度节气求解、年度朔望求解或年度农历月表生成的最小基准。
4. 让验证文档明确数据来源、误差容忍度与更新规则。

## 五、当前明确暂缓

这一阶段当前不该抢跑的内容：

- 全量历史天文数据仓库。
- 复杂的多实现矩阵与自动报告系统。
- 在核心结果还未收口前就做过深的性能优化工程。

## 六、完成判定

第五阶段完成的最低标准是：

```txt
核心时间、
太阳、
月亮、
中国历法结果
都有可复核 fixture；
conformance 至少能跑通一条链路；
benchmarks 至少覆盖一条主路径。
```
