# Epheon 后最小链路计划：从可验证内核到可对外使用

本文不是新的架构阶段，也不替代 `docs/plans/0001-0006.md`。

它只做一件事：在第一至第五阶段最小链路已经收口之后，重新定义当前主线。

当前判断：

```txt
Epheon 已经完成“最小可验证内核”，
但还没有完成“面向外部稳定使用的全量目标”。
```

也就是说：

- 现在已经可以证明核心链路能工作。
- 现在还不应该声称架构图中的所有目标都已完全落地。
- 下一步应该先补“稳定化”，再决定是否进入“扩展化”和“产品化”。

## 一、本文和现有计划的关系

保留不变：

- `docs/plans/0001-0006` 继续表达架构阶段边界。
- `docs/tasks/0001-0006` 继续维护已完成任务与暂缓任务状态。

本文新增：

- 定义最小链路收口后的新主线顺序。
- 给出“何时算可以进入全量目标”的进入条件。
- 把“现在就该做的事”和“还不该做的事”重新分开。

## 二、当前现状判断

当前已经具备：

```txt
时间模型
参考系
太阳黄经与节气
月亮位置与朔望
中国历法最小 modern 规则集
fixture / conformance / benchmark 最小闭环
```

当前仍然缺少：

```txt
更完整的对外文档与精度口径
更广的 fixture 覆盖
至少第二个真实历法实现
@epheon/calendars 统一抽象的真实落点
@epheon/compat / @epheon/runtime / @epheon/cli 的真实使用场景
对外发布与长期维护约定
```

因此，当前主线不应该直接理解为：

```txt
开始把架构图里所有 package 一次性补完
```

而应该理解为：

```txt
先把已经存在的能力打磨成稳定边界，
再根据真实需求决定哪些“最终目标”应该进入实现。
```

## 三、当前主线顺序

从现在开始，按下面顺序推进：

1. 先做稳定化，补齐最小链路的文档、覆盖和对外口径。
2. 再做扩展化，但只进入被真实需求证明的部分。
3. 最后做产品化，把内核能力变成可发布、可接入、可演示的交付物。

## 四、第一段：稳定化

这一段是新的当前主线。

目标：

```txt
把“已经能算”提升为“可以清楚说明能算到什么程度”。
```

优先实现：

1. 补公共 README 与包说明，优先清理审计里已知缺口和过时描述。
2. 扩充 `standards/`，把中国历法、节气、朔望样例从“最小证明”补到“可复核集合”。
3. 把精度、误差来源、适用边界写成明确文档，而不是散落在测试与注释里。
4. 明确 `@epheon/calendar-chinese` 当前只承诺 `modern` 规则集，不提前暗示历史规则支持。
5. 复核公开 API 边界，避免把 internal helper、实验性能力或尚未稳定的语义提前对外承诺。

这一段完成后，应该可以清楚回答：

```txt
当前实现覆盖哪些能力？
当前结果和哪些标准样例对齐？
当前 tolerance 是什么？
当前哪些能力仍然不保证？
```

## 五、第二段：扩展化

只有第一段完成后，才进入这一段。

目标：

```txt
只实现那些已经被第二个真实场景证明需要的能力。
```

优先顺序：

1. 如果先出现第二种真实历法需求，再进入：

```txt
@epheon/calendar-gregorian
@epheon/calendar-julian
@epheon/calendars
```

2. 如果先出现真实的外部接入需求，再进入：

```txt
@epheon/compat
```

3. 如果先出现真实的运行时装配或大数据加载需求，再进入：

```txt
@epheon/runtime
```

这一段的核心规则：

```txt
没有第二个真实实现，就不要抽 @epheon/calendars。
没有真实外部格式，就不要抢跑 compat。
没有真实装配压力，就不要抢跑 runtime。
```

## 六、第三段：产品化

只有第二段里至少有一条真实消费链路后，才进入这一段。

目标：

```txt
让 Epheon 从“仓库内可验证”变成“对外可使用”。
```

优先实现：

1. 设计最小 `@epheon/cli`，只提供少量只读查询命令。
2. 明确版本策略、发布流程、兼容性承诺与变更说明。
3. 补对外 Quick Start、最小示例和能力边界说明。
4. 视需要把 conformance / benchmark 接入更稳定的 CI 入口。

这一段完成后，才更接近架构文档里的“全量目标”。

## 七、当前明确不该做的事

当前不应进入主线：

- 大而全的历史历法规则支持。
- 为未来猜测预建 `@epheon/calendars` 抽象。
- 没有真实用户需求时就把 `compat`、`runtime`、`cli` 一次性全开。
- 把“已有样例验证”表述成“已经有完整精度报告”。
- 在 README 或对外介绍里暗示项目已经完成全部架构目标。

## 八、进入下一轮任务拆分的条件

本文确认后，下一步才进入新的任务拆分。

建议拆法：

1. 先新增一份“稳定化任务清单”。
2. 让任务优先覆盖：
   - README 缺口
   - fixture 覆盖面
   - 精度与边界文档
   - 对外 API 承诺收口
3. 扩展化和产品化先不拆成可执行任务，直到进入条件成立。

## 九、当前完成判定

这份新主线计划完成的标志不是“开始写新包”，而是下面这件事成立：

```txt
团队对当前仓库的共同认识已经从
“最小链路做完了，接下来随便补”
切换成
“先稳定化，再按真实需求扩展，再产品化”
```

在这之前，不建议直接把第六阶段从“暂缓”改成“进行中”。
